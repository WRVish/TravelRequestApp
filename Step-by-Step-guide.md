# Corporate Travel Request and Automation Setup Guide

This guide walks you through building an automated travel request approval system. We use a custom SharePoint SPFx Web Part to submit parent-child requests, SharePoint lists to store data, and Power Automate to handle manager approvals, email alerts, Teams cards, PDF generation, and runtime logging.

---

## 1. Relational Data Mapping Strategy

To capture travel requests, we use a parent list for the trip details (purpose, destination, dates) and a child list for estimated costs (flights, accommodation, meals):

```text
[CLIENT FORM SUBMISSION]
User submits Travel Request ──► SharePoint generates Internal ID (e.g., 205)
                                    │
                                    ▼
User submits Estimated Costs ──► Sets TravelRequest (Lookup Field) = 205

[POWER AUTOMATE FLOW]
Flow triggers on Parent ──► Generates 'TRV-2026-205' and updates Parent
                                    │
                                    ▼
Queries Child Rows where TravelRequestId eq 205 ──► Updates Child rows with 'TRV-2026-205'
```

1. **Submission Phase:** The traveler fills the SPFx form. The parent record is saved first. Once the parent `ID` is returned (e.g. `205`), the details list records are saved with their lookup column (`TravelRequest`) pointing to that parent ID.
2. **Propagation Phase:** The flow triggers on the new parent request, generates the clean `TravelID` string (e.g. `TRV-2026-205`), updates the parent, finds the linked child rows, and updates their `TravelID` fields.

> [!IMPORTANT]
> **Manual Lookup Column Creation Required:**
> Because SharePoint does not support creating Lookup columns directly during a CSV import, the column `TravelRequest` is **excluded** from the [Travel Request Details.csv](./assets/Travel%20Request%20Details.csv) file.
> 
> Follow these steps to provision this list:
> 1. Create/import the list using [Travel Request Details.csv](./assets/Travel%20Request%20Details.csv). This will create all columns except `TravelRequest`.
> 2. Go to the settings for the `Travel Request Details` list, click **Add column**, and select **Lookup**.
> 3. Name the column **`TravelRequest`**.
> 4. In the column settings, select **`Travel Request`** as the source list and choose **`ID`** as the field to display. Make sure it is marked as **Required**.

## 3. SPFx Web Part Build & Deployment Guide

Follow these instructions to compile, package, and deploy the Custom Travel Request SPFx web part:

### 3.1 Restoring Dependencies
1. Open your terminal and navigate to the project directory:
   ```bash
   cd TravelRequestApp
   ```
2. Run the following command to restore dependencies:
   ```bash
   npm install
   ```

### 3.2 Building & Packaging
The project uses the **Heft** build toolchain profile:
1. Compile the web part source code in production mode:
   ```bash
   npm run build
   ```
   *This command executes `heft build --clean --production` and then packages the solution to produce `TravelRequestApp/solution/custom-travel-request.sppkg`.*

### 3.3 Deployment to SharePoint
1. Navigate to your SharePoint Online Admin Center, then open the **Tenant App Catalog** (or Site Collection App Catalog).
2. Upload the solution file located at `TravelRequestApp/solution/custom-travel-request.sppkg` to the catalog.
3. Choose to deploy the solution and click **Enable App**.

### 3.4 Web Part Mappings Configuration
1. Go to any modern SharePoint Page, edit the page, and add the **Custom Travel Request** web part.
2. Edit the web part properties. In the property pane:
   * **Select Travel Requests List (Parent):** Choose your `Travel Request` list.
   * **Select Details List (Child):** Choose your `Travel Request Details` list.
   * Map the respective columns for both lists based on the column dropdown options loaded from list schemas.
   * *Note: The Approver is written directly to the standard list text column named `Approver`. You do not need to configure an Approver mapping column in the web part properties; fallback options are configurable via the `Fallback Approvers` People Picker field in the web part property pane.*
3. Save and publish the page.

## 4. Power Automate Step-by-Step Flow Blueprint

Follow these steps to build the processing and approval sequence. This maps the flow layout shown in the architectural image exactly.

```text
[Flow Trigger: When an item is created]
       │
[Step 2 — Initialize Variables] (RequestID, ManagerEmail, Status)
       │
[Step 3 — Get Manager Details] (Get manager of the requester)
       │
[Step 4 — Travel ID Construction & Variable Updates]
       │
[Step 5 — Create an Approval (Start and wait)] (Approver: Manager)
       │
[Step 6 — Condition: Check Approval Outcome]
 ├── YES (Approved Branch)
 │    ├── [5.1 Update item status] (Set status to 'Approved' and update parent list)
 │    ├── [5.2 Fetch child details & admin branding configuration]
 │    ├── [5.3 Create HTML document & convert to PDF via OneDrive]
 │    ├── [5.4 Create final PDF in SharePoint Online]
 │    ├── [5.5 Send approval email to requester with PDF attached]
 │    └── [5.6 Post approval message in Teams]
 │
 └── NO (Rejected Branch)
      ├── [5.1 Update item status] (Set status to 'Rejected' and update parent list)
      ├── [5.2 Send rejection email with comments]
      └── [5.3 Post rejection message in Teams]
       │
[Step 7 — Log Approval Details] (Outside condition: Store approver, date, comments, status in SharePoint list)
       │
[Step 8 — Send Final Confirmation Email to Requester] (Outside condition: With status and comments)
       │
[Step 9 — End / Terminate Succeeded]
```

---

### Step 1 — Initialize Trigger
1. Access the Power Automate portal, select **Create** -> **Automated cloud flow**.
2. Set the flow name to: `TravelRequest-ApprovalFlow`.
3. Select trigger: **SharePoint — When an item is created**.
4. Configure the trigger parameters:
   - **Site Address:** Paste your SharePoint site URL.
   - **List Name:** Select `Travel Request`.

---

### Step 2 — Initialize Variables
Add three **Initialize variable** actions directly under the trigger block:
1. **Name:** `RequestID`
   - **Type:** Select `String`.
   - **Value:** *Leave empty.*
2. **Name:** `ManagerEmail`
   - **Type:** Select `String`.
   - **Value:** Bind the approver email submitted by the SPFx form: `triggerOutputs()?['body/Approver']` *(Since the SPFx form already pre-fills the manager or falls back to the configured dropdown value, we bind it directly here to handle cases where a user has no manager).*
3. **Name:** `Status`
   - **Type:** Select `String`.
   - **Value:** *Leave empty.*

---

### Step 3 — Get Manager Details (Optional Backup)
*Note: Since we initialized `ManagerEmail` directly from the SharePoint item's `Approver` field, this step serves as a backup check in case the Approver value was blank.*
1. Add a **Condition** card to check if `ManagerEmail` is empty.
2. If empty, add an Office 365 Users action card: **Get manager (V2)**.
   - **User (UPN):** Bind the traveler's email: `triggerOutputs()?['body/TravelerEmail']`.
3. Inside the condition, add a **Set variable** action:
   - **Name:** Select `ManagerEmail`.
   - **Value:** `outputs('Get_manager_(V2)')?['body/mail']`.

---

### Step 4 — Travel ID Construction & Parent ID Update
1. Add a **Compose** block named `cmp TravelID` to generate the custom corporate request ID:
   ```powerautomate
   concat('TRV-', formatDateTime(utcNow(), 'yyyy'), '-', triggerOutputs()?['body/ID'])
   ```
2. Add a **Set variable** action:
   - **Name:** Select `RequestID`.
   - **Value:** Choose the output content of the `cmp TravelID` compose block.
3. Add a SharePoint **Update item** action for `Travel Request` to write this ID back to the list:
   - **Id:** `triggerOutputs()?['body/ID']`
   - **Title:** `triggerOutputs()?['body/Title']`
   - **TravelID:** `variables('RequestID')`
   *(Leave all other fields blank to keep traveler-submitted values).*

---

### Step 5 — Create an Approval (Start and wait)
1. Add a **Start and wait for an approval** action card:
   - **Approval Type:** Select `Approve/Reject - First to respond`.
   - **Title:** Enter: `Travel Request Approval Request - @{variables('RequestID')}`
   - **Assigned To:** Bind the manager's email variable: `variables('ManagerEmail')`
   - **Details:** In the details text box, paste traveler details for review:
     ```text
     A travel booking request requires your administrative review:
     
     - **Traveler:** @{triggerOutputs()?['body/TravelerName/DisplayName']}
     - **Travel ID:** @{variables('RequestID')}
     - **Destination:** @{triggerOutputs()?['body/Destination']}
     - **Departure Date:** @{formatDateTime(triggerOutputs()?['body/FromDate'], 'yyyy-MM-dd')}
     - **Return Date:** @{formatDateTime(triggerOutputs()?['body/ToDate'], 'yyyy-MM-dd')}
     - **Estimated Total Cost:** @{triggerOutputs()?['body/EstimatedCost']}
     ```

## 5. Try-Catch Scope Error Handling

Group your variables, get manager, get items, loop, approvals, and branching blocks inside two Scopes:
1. **Scope_MainProcessing:** Place variables generation, get manager, get items, loop, approvals, and branching blocks inside this scope.
2. **Scope_ErrorHandler:** Place a SharePoint **Create item** inside this scope targeting `Flow Error Logs` list:
   - **Title:** `Flow runtime error caught`
   - **FlowName:** `TravelRequest-ApprovalFlow`
   - **TravelID:** `variables('RequestID')`
   - **ErrorMessage:** `actions('Scope_MainProcessing')?['error']?['message']`
   - **Timestamp:** `utcNow()`
3. Configure `Scope_ErrorHandler` **Configure Run After** settings to run only if `Scope_MainProcessing` **has failed**, **is skipped**, or **has timed out**.
